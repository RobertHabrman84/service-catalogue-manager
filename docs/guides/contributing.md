# Contributing Guide

## Welcome

Thank you for your interest in contributing to Service Catalogue Manager!

## Getting Started

1. Read the [Development Setup Guide](./development-setup.md)
2. Review [Coding Standards](./coding-standards.md)
3. Understand our [Git Workflow](./git-workflow.md)

## How to Contribute

### Reporting Bugs

1. Check existing issues first
2. Create detailed bug report:
   - Steps to reproduce
   - Expected behavior
   - Actual behavior
   - Environment details

### Suggesting Features

1. Check existing feature requests
2. Create feature request with:
   - Problem description
   - Proposed solution
   - Alternatives considered

### Code Contributions

1. Fork the repository
2. Create feature branch
3. Make changes
4. Write/update tests
5. Submit pull request

## Pull Request Process

### Before Submitting

- [ ] Code follows standards
- [ ] All tests pass
- [ ] Documentation updated
- [ ] No merge conflicts
- [ ] Linked to work item

### PR Template

```markdown
## Description
Brief description of changes

## Type of Change
- [ ] Bug fix
- [ ] New feature
- [ ] Breaking change
- [ ] Documentation

## Testing
How was this tested?

## Checklist
- [ ] Tests added/updated
- [ ] Documentation updated
- [ ] No breaking changes
```

### Review Process

1. Automated checks run
2. Reviewer assigned
3. Feedback addressed
4. Approval received
5. Merge to develop

## Code Review Guidelines

### For Authors

- Keep PRs small and focused
- Respond to feedback promptly
- Be open to suggestions
- Explain complex changes

### For Reviewers

- Be constructive
- Explain reasoning
- Approve when ready
- Follow up on changes

## Development Guidelines

### Commit Messages

```
feat(component): add new feature

- Detail 1
- Detail 2

Closes #123
```

### Branch Naming

```
feature/SCM-123-short-description
bugfix/SCM-456-fix-something
```

### Testing

- Unit tests for business logic
- Integration tests for APIs
- E2E tests for critical paths

## Documentation

- Update README if needed
- Add JSDoc/XML comments
- Update API documentation
- Add ADR for decisions

## Community

### Communication

- GitHub Issues for bugs/features
- Slack for discussions
- Weekly sync meetings

### Code of Conduct

- Be respectful
- Be inclusive
- Be collaborative
- Be professional

## Recognition

Contributors are recognized in:
- CHANGELOG
- Release notes
- README contributors section
